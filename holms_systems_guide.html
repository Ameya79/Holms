How Holms Works: Complete Technical Guide

A comprehensive, function-by-function architectural blueprint of Holms — a privacy-first, single-tenant local document search engine and RAG assistant. Covers ingestion, OCR, FastEmbed vectors, SQLite FTS5 + sqlite-vec, RRF ranking, multi-provider LLM synthesis, and Next.js frontend binding.

00 — Orientation

Product Mental Model & Principles

Holms is designed under a single-tenant, local-first architecture per PRD §2. Every installation of Holms runs on the user's own machine. Your files, document chunks, FTS index, and vector embeddings reside strictly inside data/index.db on local disk.

The Core Architectural Promise: Holms operates in two distinct operational modes without requiring multi-tenant cloud authentication or external storage databases:
  1. Search-Only Mode (Zero-Cost / No API Key): Ingestion, OCR, FastEmbed vectors, FTS5 keyword indexing, and hybrid retrieval run 100% locally. Returns ranked document cards with snippet highlights directly without calling any remote API.
  2. AI Answer Mode (BYO Key): Performs a single LLM call (Anthropic Sonnet 3.7 / Claude 3.5, Gemini 2.0 Flash, OpenAI GPT-4o-mini, or Groq Llama 3.3) to verify retrieved candidate chunks and synthesize a grounded natural language answer with document source citations.

03 — Ingestion Pipeline

File Ingestion & Silent OCR Pipeline

The ingestion module (backend/ingestion.py) processes incoming .pdf, .docx, .png, .jpg, .txt, and .md files without requiring manual user intervention.

1. Text Extraction & Native Density Check

When a PDF page is parsed by extract_text_from_pdf(), native text length is inspected. If a page yields fewer than 20 characters of text, Holms flags the page as scanned/image-based and routes it to OCR:

# backend/ingestion.py - PDF text extraction & OCR trigger
for page in doc:
    text = page.get_text().strip()
    if len(text) < 20:   # Threshold check from TRD §3
        ocr_used = True
        pix = page.get_pixmap(dpi=300)   # Render high-res 300 DPI buffer
        img = Image.frombytes("RGB", [pix.width, pix.height], pix.samples)
        text = pytesseract.image_to_string(img)
    full_text.append(text)

2. Sentence-Aware Chunking Algorithm

Rather than splitting blindly by character count (which ruptures sentences and ruins embeddings), chunk_text() uses Regex boundary splitting re.split(r'(?<=[.!?])\s+', text) to build sentence groups up to ~300 words with a 1-sentence overlap:

# backend/ingestion.py - Sentence-aware chunking
def chunk_text(text: str, target_words: int = 300, overlap_sentences: int = 1) -> list[str]:
    sentences = split_sentences(text)
    chunks, current = [], []
    current_len = 0
    for sentence in sentences:
        words = len(sentence.split())
        if current_len + words > target_words and current:
            chunks.append(" ".join(current))
            current = current[-overlap_sentences:]   # Overlap carry-forward
            current_len = sum(len(s.split()) for s in current)
        current.append(sentence)
        current_len += words
    if current:
        chunks.append(" ".join(current))
    return chunks
04 — Vector Engine & Database

Embeddings & SQLite Hybrid Storage

Holms utilizes fastembed with the BAAI/bge-small-en-v1.5 model running ONNX CPU inference. Model initialization happens once on startup (~130 MB ONNX weights file cached locally).

Asymmetric BGE Embedding Rule: BGE models require an instruction prefix on search queries to optimize asymmetric retrieval, but document chunks must be embedded as raw plain text. 
# backend/embeddings.py - Asymmetric embedding handling
def embed_documents(chunks: list[str]) -> list[np.ndarray]:
    return list(_embedder.embed(chunks)) # Plain text without prefix

def embed_query(query: str) -> np.ndarray:
    prefixed = f"Represent this sentence for searching relevant passages: {query}"
    return list(_embedder.embed([prefixed]))[0]

Database Schema (data/index.db)

The database initialization in backend/database.py configures WAL mode (Write-Ahead Logging) and constructs four interconnected tables:

-- 1. Document metadata master table
CREATE TABLE documents (
    id TEXT PRIMARY KEY,
    filename TEXT NOT NULL,
    filepath TEXT,
    uploaded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    ocr_used BOOLEAN DEFAULT 0
);

-- 2. Chunk text & raw binary embedding store
CREATE TABLE chunks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    doc_id TEXT NOT NULL,
    chunk_index INTEGER NOT NULL,
    text TEXT NOT NULL,
    embedding BLOB,  -- Float32 array serialized via struct.pack / numpy.tobytes
    FOREIGN KEY(doc_id) REFERENCES documents(id) ON DELETE CASCADE
);

-- 3. SQLite FTS5 Full-Text Search virtual table (BM25 keyword index)
CREATE VIRTUAL TABLE chunks_fts USING fts5(
    text,
    content='chunks',
    content_rowid='id'
);

-- 4. sqlite-vec extension vec0 virtual table (Dense vector KNN search)
CREATE VIRTUAL TABLE vec_chunks USING vec0(
    id INTEGER PRIMARY KEY,
    embedding float[384]
);
05 — Retrieval Engine

Hybrid Search & Reciprocal Rank Fusion (RRF)

Pure vector search fails on exact filenames or serial numbers, while pure keyword search fails on semantic intent ("what did I pay last month"). Holms runs both search paradigms in parallel and fuses their rankings using Reciprocal Rank Fusion (RRF).

1. BM25 Keyword Retrieval via FTS5

search_bm25() quotes query terms and queries SQLite's built-in bm25(chunks_fts) function to return top 60 sparse matches.

2. Vector KNN Retrieval & NumPy Fallback

search_vector() first attempts high-speed SIMD vector search using sqlite-vec's vec_chunks table. If the binary extension is unavailable on the OS, it falls back to _numpy_brute_force() which computes cosine similarity across all chunk blobs using vector dot products:

# backend/search.py - Cosine similarity matrix multiplication fallback
query_norm = np.linalg.norm(query_vec)
vec_norms = np.linalg.norm(vectors, axis=1)
sims = (vectors @ query_vec) / (vec_norms * query_norm)
top_k_indices = np.argsort(-sims)[:k]

3. Reciprocal Rank Fusion (RRF) Math

RRF merges rankings by position score rather than raw distance metrics (which live on incompatible scales):

$$\text{RRF\_Score}(d) = \sum_{r \in \text{Rankers}} \frac{1}{k_{const} + \text{rank}_r(d)}$$ Where $k_{const} = 60$. A document ranked #1 in BM25 and #3 in Vector search receives: $$\text{Score} = \frac{1}{60 + 1} + \frac{1}{60 + 3} = 0.01639 + 0.01587 = 0.03226$$ 
# backend/search.py - Reciprocal Rank Fusion implementation
def reciprocal_rank_fusion(bm25_results, vec_results, k_constant=60):
    scores = {}
    for rank, res in enumerate(bm25_results):
        cid = res["id"]
        scores[cid] = scores.get(cid, 0.0) + 1.0 / (k_constant + rank + 1)
    for rank, res in enumerate(vec_results):
        cid = res["id"]
        scores[cid] = scores.get(cid, 0.0) + 1.0 / (k_constant + rank + 1)
    return scores

07 — Client Architecture

Next.js Frontend & UI Integration

The client is built with Next.js 15 App Router using client components and static export capabilities (`output: 'export'`). Data communication between Next.js and FastAPI occurs via `lib/api.ts` fetch helpers.

Component Breakdown & Data Flow

  • app/app/page.tsx: Main container state orchestrator holding query, documents, answer, loading, and isSettingsOpen.
  • components/TopBar.tsx: Header displaying connection status and triggering settings modal or upload file dialog.
  • components/SearchBox.tsx: Input bar supporting keyboard submit (Enter key), loading spinner, and quick pre-filled prompt pills.
  • components/DocumentCard.tsx: Visual result card displaying file type badges (PDF DOCX IMG TXT), file size, relevance percentage match score, highlighted snippet, and collapsible chunk inspector.
  • components/AnswerBox.tsx: Container rendering AI synthesis answer markdown with document source citations.
  • components/UploadZone.tsx: Bottom bar for file drag-and-drop uploads, showing real-time indexing status feedback.
  • components/SettingsModal.tsx: Provider selector and API key manager with live connection testing via /settings/test.

09 — Operations

System Startup & Build Order

  1. Environment Validation (`run.py`): Checks Python 3.10+, locates Tesseract executable on PATH or Windows Program Files, tests `sqlite-vec` extension import.
  2. Database Bootstrapping (`init_db()`): Enables WAL journal mode, creates tables, and builds FTS5 + vec0 virtual indexes.
  3. Model Pre-Warming (`embed_query("warmup")`): Loads ONNX `bge-small-en-v1.5` model into RAM so initial user queries execute sub-second.
  4. Server Binding: Binds Uvicorn ASGI server to local host loopback `127.0.0.1:8000`.
Command to launch Holms: 
python run.py
